# Generator Design

> Status: **built.** The 3-stage concept→code pipeline + quality gate are implemented
> (built grader-first — see [`grader_design.md`](./grader_design.md)). This doc records the
> generation decisions; the first entry is a constraint **inherited from the grader**.

## Inherited constraint: the generator owns the font palette

The grader renders both the reference and the candidate in **one sealed, offline,
font-fixed environment** so that the same HTML always becomes the same picture — the
*deterministic offline render + bundled fonts* principle (see
[`grader_design.md` → "Principle: deterministic offline render + bundled fonts"](./grader_design.md)).

**The open dependency.** Today the render/verifier image bundles only **DejaVu**
(`apt-get install fonts-dejavu-core` in the emitted verifier `Dockerfile`). That is
*deterministic* — both sides fall back identically, so grading stays fair — but it is **not
faithful**: if a generated site's CSS asks for a font the image lacks, the browser silently
substitutes DejaVu, so the design's intended typography never actually appears (for the
agent to replicate *or* for the reference). The grader cannot fix this on its own; **the
generator must.**

**What the generator must do (the fix).**

1. **Define a curated font palette** — a small set (~5–10 families), all open-licence and
   visually diverse enough to give real typographic variety across generated sites.
2. **Bundle exactly that palette into the render/verifier image** — install the font files in
   the `Dockerfile` (and/or ship them and reference via a relative `@font-face src` served
   over the local HTTP server, as `render/browser.py`'s docstring sketches).
3. **Constrain generation to that palette** — the generator's prompts/templates may only use
   `font-family` values from the palette, and must **not** pull Google Fonts or any external
   webfont (the offline render blocks those anyway, so an external font would just fall back
   and look wrong).
4. **One source of truth** — the palette list is shared by *(a)* the generator's allowed
   fonts and *(b)* the image's installed fonts. They must stay in sync, or determinism vs.
   fidelity diverges again.

Result: typography becomes both **faithful** (renders as the design intends) **and
deterministic** (identical everywhere the image runs — Docker locally, Modal in the cloud).

**Related (same root cause): assets/images.** External/CDN assets have the identical
problem — blocked by the offline render, so they'd render empty. The generator should prefer
**CSS-drawable** visuals (gradients, shapes, icon fonts) or **locally-bundled** assets so the
offline render is faithful. Tracked as its own generation decision below.

## Generation pipeline: 3 stages (initial)

A site is generated by a **multi-stage** pipeline, not a single call. Single-shot
"generate a 5-page site" mode-collapses (every site becomes the same template) and
drifts (each page re-invents the styling). The pipeline's shape follows one principle:

> **Separate global state from per-page state.** Brand, design system, navigation and
> shared components are decided **once** and *frozen*; pages are generated **per-page**
> against that frozen state. Consistency lives in shared *artifacts* (real CSS files),
> never in shared *descriptions* (prose an LLM re-interprets per page).

| Stage | Name | Owns | Output | Execution |
|---|---|---|---|---|
| 1 | **Big picture** | the *what* | brief + sitemap (+ slugs → `page_map`) + per-page section list + **component manifest** | 1 call, high-temp (diversity injection) |
| 2 | **Design system** | the *how it looks* | `variables.css` (tokens) + `components.css` (recipes) + header/footer partials | **1 coherent call** |
| 3 | **Pages** | the *assembly* | one HTML file per page, composing the frozen system | **fan out, parallel per page** |

**Stage 1 — big picture / diversity seed.** Turns a *sampled seed* (industry × aesthetic
× audience × mood, drawn in code, not by the LLM) into a brand brief, a sitemap (≥5
pages), the section list per page, and a **component manifest** — the deduplicated union
of every section/component type any page needs. Diversity comes from the sampled inputs,
not the model's variance (otherwise: coffee shops and yoga studios forever). No code yet.

The sitemap is the **single source of truth for page identity and substance.** Each page
gets a stable **slug** (collision-safe; home → `index.html`) from which *everything*
downstream derives — the reference screenshot name (`about.png`), the expected agent file
(`about.html`), and the `page_map = {slug: {screenshot, expected_file}}` (which is just the
sitemap frozen, not separately authored). And the **per-page section list *is* the
substance budget**: stage 1 decides "enough content" by giving each page a real section set
(a Contact page gets more than an address). Downstream stages and the gate **verify** this
intent survived to pixels — they don't re-decide it.

**Stage 2 — design system as frozen code.** Authors the shared artifacts *as real CSS*,
not prose:
- `variables.css` (`:root` tokens) locks **values** — palette, type scale, spacing,
  radius. Stops `#2563eb`-vs-`#2f6ae0` value drift.
- `components.css` locks **structure** — `.btn`/`.card`/`.nav` etc. as authored recipes.
  Variables alone don't prevent structural drift: two pages can honor the same tokens yet
  assemble a button with different padding/weight. The recipe must be frozen too.
- header/footer/nav ship as **byte-identical injected partials**, not re-derived per page.
- Must satisfy **every entry in the manifest** — a missing component forces stage 3 to
  improvise un-styled CSS, which is exactly the drift we're preventing.
- This is **one call** (not fanned out): components must be mutually coherent — a button,
  card and input share the same radius/shadow language — so they're designed *together*.

**Stage 3 — per-page fan-out.** Parallel, one call per page, each reading the *frozen*
stage-2 artifacts read-only. Its job is **composition**: emit semantic markup using
existing component classes + inject the chrome partials. The only fresh CSS it writes is
**page-specific layout** (grid, gaps) and it may reference **only** declared `var(--…)`
tokens — never a new color, size, or raw px. Pages are consistent *by construction*
(same stylesheet, same partials), not by the model remembering to be consistent.

**Why parallelism differs by stage:** the consistency requirement decides it, not the
stage count. Anything that must be mutually coherent stays in **one call** (stage 2);
anything independent **fans out** (stage 3).

**Open dials (decide later):**
- *Content location* — copy in stage 1 (pure-renderer stage 3, max control) vs in stage 3
  (more natural variety). Current lean: stage 1 fixes section structure + key headlines;
  stage 3 fleshes out body copy within it.
- *Component-set size* — small (more page variety, more drift surface) vs rich (near-zero
  drift, stage 3 becomes pure composition). Lean: freeze everything that *repeats within a
  site*, leave section layout free.

**Stages 4+ are the quality gate** (below).

## Agent-task constraints the generator must honor (well-posedness)

The generated site is the **stimulus the coding agent (Opus) replicates** — *not* an
artifact produced to satisfy the grader. The agent only ever sees **screenshots rendered
in the sealed env**, then writes HTML/CSS to reproduce them. So a target is a *fair,
well-posed task* only if everything the agent must reproduce is **visible in, and
recoverable from, that screenshot** — and **faithfully re-renderable** in the same env.
These constraints are about the task being *possible*, and are independent of how grading
works.

1. **Faithful, deterministic render is the floor.** The picture the agent studies must be
   *exactly* reproducible in the same sealed env, or a perfect replica is impossible by
   construction. This is why **hermeticity** (no external resources), the **pinned offline
   engine**, and the **bundled font palette** (see the inherited-constraint section up top)
   are load-bearing — and why the **font palette is blocking for generation**, not a
   nicety: until it lands, the screenshots the agent must match are font-substituted to
   something the design never intended.
2. **One fixed 1280px desktop, full-page capture.** The agent infers the entire design
   from a single fixed-width *static* screenshot. So the design must be **legible and
   complete at 1280px** — avoid `100vh`-dependent heroes, responsive-only layouts whose
   content only resolves at other widths, or anything whose information lives off that one
   frame.
3. **Static for Part 1.** No animation, hover/focus-only, scroll-triggered, or randomized
   state — a static capture cannot convey it, so the agent cannot recover it. (Motion is
   Part 2, with its own capture protocol.) The generator must emit static Part-1 sites.
4. **Target identity / `page_map` contract.** The agent is shown N screenshots and must
   emit files; the grader pairs candidate↔reference via `page_map = {slug: {screenshot,
   expected_file}}`, and a page that doesn't line up scores **0**. This is **derived from
   stage 1's sitemap, not separately authored**: each page's slug names its screenshot
   (`about.png`) *and* the expected agent file (`about.html`) — collision-safe, home →
   `index.html`. The **task tells the agent exactly which filenames to produce**, so its
   output aligns by construction.
5. **Per-page substance.** Each of the ≥5 pages must carry enough real design content to be
   a meaningful replication challenge — near-empty pages make a **weak task** (nothing to
   reproduce; per-page-mean grade diluted). This floor is **owned by stage 1** (the
   per-page section list is the substance budget); stage 4 only **verifies the render
   realized it**. "≥5 pages" is necessary but not sufficient.

> Deliberately **out of scope here:** designing the generator around grader-metric quirks
> (OCR legibility, palette-area dominance, SSIM saturation on minimal designs). We generate
> the **full, realistic distribution** the brief asks for; where the grader discriminates
> more softly on some aesthetic is a finding for the **validation report**, *not* a reason
> to bias generation or reject those sites.

## Quality gate (stages 4+)

The gate asks **"is this a *valid, faithfully-replicable target* for the agent?"** — it
enforces the well-posedness constraints above plus the consistency invariants from stages
1–3. The gate is a **cheap → expensive funnel**: kill obvious garbage with deterministic
checks before spending a render or an LLM judge. A failure either **rejects** the site or
triggers **targeted repair** (off-token color → fix that file; one broken page →
regenerate just that page).

**V1 = stage 4 + stage 5** — both deterministic/objective (no judge variance, no API
cost, fully reproducible), they catch the bulk of garbage, and they are the *enforcement
teeth* behind the consistency invariants stages 1–3 are built around.

### Stage 4 — structural + consistency + hermeticity (deterministic, ~free)

- **Completeness** — all sitemap pages actually generated (the ≥5 *count* is gated inline
  right after stage 1 — see *Failure handling* below); HTML and CSS parse; shared files
  (`variables.css`, `components.css`, partials) exist and link-resolve.
- **Target identity** — the canonical page set + fixed filenames + `page_map` are emitted,
  and every reference page maps 1:1 to an expected agent filename (constraint 4 above).
- **Per-page substance** — each page clears a content-density floor (not near-empty), so
  every page is a real replication challenge (constraint 5 above).
- **Static-only (Part 1)** — no animation/transition/scroll-or-hover-triggered/randomized
  state in the emitted site (constraint 3 above); these can't survive a static capture.
- **Token compliance** — scan every CSS/inline value; reject any hex/rgb/px not traceable
  to a `var(--…)`. (the anti-drift validator)
- **Manifest compliance** — every section a page references resolves to a stage-2
  component; no improvised un-styled sections.
- **Chrome identity** — header/footer/nav byte-identical across pages.
- **Hermeticity (self-containment)** — *the site must depend on nothing outside its own
  bundle*, so the sealed offline render is faithful (same root cause as the inherited
  font bug above) **and** the emitted task stays portable (Docker ↔ Modal):
  - **External *resource* loads → hard fail.** No external fonts (Google Fonts `<link>`,
    `@import`, `@font-face src: url(http…)`), images (`<img src=http…>`, CSS
    `url(http…)`), or CSS/JS (CDN `<link>`/`<script>`). Each renders empty/wrong in the
    offline env. Repair to a local or CSS-drawable equivalent, else reject.
  - **Internal refs → must resolve.** Every internal `href`/`src` (and local asset) points
    to a file that exists — no broken/dead links. (`#anchor` → element `id`: warn, not
    fail.)
  - **External *hyperlinks* (`<a href=http…>`) → allowed but inert.** The browser never
    fetches an anchor at render time, so they don't break faithfulness or determinism, and
    they keep sites realistic. Flag/log, do not fail.

### Stage 5 — render validity (uses the live render module)

- **Renders clean at 1280px** — every page renders full-page at the fixed 1280px desktop
  viewport (constraint 2) with no console/JS error.
- **Deterministic** — render twice → identical output (the *deterministic offline render +
  bundled fonts* principle). This is the honest meaning of an "oracle" check: it proves a
  perfect replica is *achievable* in the sealed env, i.e. the task is well-posed.
- **Not blank** — content actually fills the viewport; reject near-empty whitespace pages.
- **No catastrophic layout** — no horizontal overflow, zero-height sections,
  clipped/overlapping elements, or content rendered off-screen.

### Failure handling / repair policy

Repair is **localized**, not a whole-site re-roll. Two mechanisms:

**Fail-fast inline gates** — each cheap invariant is checked *immediately after the stage
that owns it*, so a doomed site dies before paying for downstream calls:
- **After stage 1:** sitemap has **≥5 pages** and the component manifest is well-formed.
  On fail, re-run **stage 1 only** (cheapest possible re-roll).
- **After stage 2** *(recommended):* **manifest compliance** — stage 2 styled a component
  for every manifest entry. This is the one failure stage-3 nudging *can't* fix cleanly (a
  page can't invent a missing component's styling without drifting), so catch it here. On
  fail, patch/re-run stage 2 before any page is generated.

**End-gate repair (stage 4/5) = nudge stage 3.** Page-level defects are fixed by
re-invoking the **failing page's** generation call with the *exact* validator error
appended (the machine diagnostic, e.g. *"`.hero` uses `#333` (not a token); link
`/team.html` does not exist"* — never a generic "make it better"), and iterating **until
it passes**. Constraints on the loop:
- **Bounded:** cap at *N* nudges per page. *Until fixed* needs a backstop or a stubborn
  page loops forever and burns tokens.
- **Composition-only:** the nudge reads the *frozen* stage-2 artifacts and may only
  recompose them — repairing one page can never re-author the design system, or the other
  pages silently drift.
- **On budget exhaustion → drop the site.** The gate is a *filter on a stream*, not a
  fix-everything contract: with many sites generated and only the good ones needed, a fresh
  seed is cheaper than fighting a marginal one. Mechanical/unambiguous violations (a value
  that exactly equals a token, an allowed-but-inert external `<a>`) are fixed/ignored in
  code with no LLM call.
- **Log every drop with its reason** — no silent discards. A systematic failure mode (e.g.
  40% failing hermeticity) is an upstream stage-1–3 *prompt* bug to fix, not attrition to
  absorb.

### Deferred (fast-follows, not V1)

- **Stage 6 — realism / quality judge (LLM).** Looks human-made (hierarchy, alignment,
  whitespace) not template/slop; styling matches the seeded archetype; copy is coherent,
  on-topic, non-lorem, non-duplicated across pages. This is about the site being a *good,
  realistic replication target* (the brief's "complex, human-like, varied"), not about
  grading — most expensive + fuzziest, so it sits behind the deterministic V1 checks.

> **Not a generation gate:** grader *discriminability* on a given target (does reward move
> monotonically under perturbation) is characterized in the **validation report**, not used
> to accept/reject generated sites. Designing generation around the grader's blind spots
> would shrink the distribution the brief is judging. We generate the full distribution;
> the grader is evaluated separately for robustness to it.

## Build sequence: one site locally, then Modal at scale

**Prove the recipe on a single site before paying for fan-out.** Mirroring the grader's
"spike on one page, then scale" discipline:

1. **Local single-site spike.** Generate **one** site end-to-end locally — stage 1→2→3 →
   gate (4/5) + repair → emit task — and **eyeball the quality**: does it look human-made
   and on-brief, render clean at 1280px, stay consistent across its ≥5 pages, survive the
   gate? This is the cheap check that the *pipeline itself* produces good targets. Only when
   a single site clears this do we parallelize — otherwise we'd fan out a broken recipe and
   pay N× to discover it.
2. **Modal at scale** (below) — once the single-site recipe is trusted.

## Generation at scale (Modal)

The brief wants **≥10 final tasks, *chosen* to showcase distribution and complexity** — so
we **overgenerate → gate → curate**, not generate-exactly-10.

**How many to generate.** `N_generate ≈ 10 / gate_yield × coverage_oversample`. With a
gate pass-rate of ~60–70% and enough surplus for the curated 10 to *span* the taxonomy,
that's ~**30–60 candidate sites**, not 10. The exact N falls out of the measured yield
(which is also Hard-Requirement-#1 "stable recipe" — a yield we can report).

**Stratified seeding for coverage.** Random draws won't span the space in ~10 survivors, so
the seed sampler **stratifies the taxonomy axes** (industry × aesthetic × layout-complexity
× …) to *cover* the grid — the "auditable coverage, not a random draw" principle. Each
site **records its seed tuple**, which (a) makes the batch auditable, (b) makes it
re-runnable, and (c) gives curation a cheap diversity signal (spread over axis tuples — no
embeddings). A batch-level **near-duplicate guard** (by tuple or coarse screenshot hash)
keeps the final 10 from clustering.

**Modal architecture.** The per-site pipeline is embarrassingly parallel across seeds, and
the render/gate must run in the **pinned verifier image we already emit** (it runs on
Docker/Modal via `--env`):

- **`modal.Function.map()` over the stratified seed list.** One invocation = one site
  end-to-end (gen API calls → gate → render+repair → emit task) = one container = one task
  artifact.
- **Generation *and* render both on Modal** — keeps load off the local machine (brief) and,
  by co-locating, the reference screenshots are produced in the **exact sealed env that
  grades them** (no host/container font drift — the issue-07 lesson).
- **Artifacts → a Modal Volume / object store**, keyed by seed id. Idempotent: a dropped
  site doesn't lose the others; re-runs are addressable.
- **Concurrency control** — cap Modal fan-out / add backoff so ~50 sites × ~15 calls don't
  trip Anthropic rate limits. (The one real operational gotcha.)

**Reproducibility = commit the 10, not regenerate.** LLM generation is non-deterministic
(temperature), so the recipe's stability is **yield** ("reliably produces working tasks"),
*not* byte-identical re-runs. The curated 10 become **committed fixtures** (reference
screenshots + emitted tasks in the repo, like the grader's regression sites) — that's what
gets judged and what the report references. Render/grade stay deterministic; generated
*content* does not, and that's fine.

**Curation → the final 10.** From the gated pool, select for **coverage** (spread across
aesthetic + complexity axes) and **de-duplication**; optionally include a deliberate
easy↔hard spread so the report's grader curve is legible.

**Strictly Part-1.** Scale generation is static HTML/CSS only; the Part-2 (animation) and
Part-3 (React/Solid) seams stay unbuilt.

> **Downstream, not built here:** the **eval/report harness** — run Claude Code (Opus 4.7)
> 10× per task → render each attempt → grade → visualize the spread, to evidence "higher
> reward = better replication" and surface model failure patterns. It's a first-class brief
> deliverable and the natural successor to this scale work (also a Modal job: ~10 tasks ×
> 10 rollouts). Named here so the seam is visible; designed separately.

## V1 decisions (grilled, 2026-05-31)

Concrete choices resolved in a grill session, enough to write V1 issues against. Each
starting number is calibratable on the local single-site spike + first batch.

| # | Decision | Choice |
|---|---|---|
| 1 | **Taxonomy axes** | **Stratified (coverage-driving):** `archetype` (~10) × `aesthetic` (~10) × `layout-complexity` (3: low/med/high). **Free modifiers** (sampled, not stratified): `audience/region`, `brand-mood`. Coverage grid = archetype×aesthetic×complexity; record each site's seed tuple. |
| 2 | **Page count** | Variable **[5, 10]**, **coupled to complexity** (low → 5–6 … high → up to 10). |
| 3 | **Content location** | **Stage 1** owns *naming/anchoring* copy (page/section headings, nav labels, CTAs, tier names, brand). **Stage 3** owns *filling* copy (paragraphs, blurbs, list items, captions). |
| 4 | **Component set** | **Per-site subset from a canonical catalog** (below). Stage 1's manifest selects; stage 2 authors *exactly* that subset, styled for the aesthetic. |
| 5 | **Asset/imagery** | **CSS-drawable + solid-color placeholder blocks + SVG patterns only. No raster assets** (keeps the task perfectly well-posed — the agent can reproduce 100% of what it sees). Photo-assets-shipped is a post-V1 upgrade. |
| 6 | **Font palette** | 8 OFL families + DejaVu fallback: Inter, Work Sans, Space Grotesk, Archivo(+heavy/Anton), Playfair Display, Source Serif 4, Poppins, JetBrains Mono. **Display faces headings-only.** **OS-level install** (vendored `.ttf` → `/usr/share/fonts` + `fc-cache`, referenced by bare family name — mandatory for agent well-posedness). **Render agent screenshots in the same sealed image** → subsumes issue 09 (no `@font-face`, no host render). One **fonts manifest** → image install + generator allow-list + gate hermeticity check. |
| 7 | **Per-page substance floor** | Page passes iff **≥3 distinct catalog components** (excl. header/footer) **AND ≥50 words** real text **AND rendered height ∈ [600px, 12000px]**. Measured DOM + render; verifies stage-1 intent. |
| 8 | **Static-only enforcement** | **Hard-fail:** any `<script>`; `@keyframes`/`animation:`; interaction-*reveal* rules (`:hover/:focus/:target/:checked` toggling `display`/`visibility`/`opacity`/`max-height` of content). **Allowed:** `transition`, cosmetic hover. Disclosure components (faq/accordion/tabs) render **open**. |
| 9 | **Slug rules** | **Home → `index`** (only home may); else `slugify(title)` (lowercase, ASCII-fold, hyphenate, ≤40 chars), collisions → numeric suffix. Slug is the single identity token → `<slug>.html` + `<slug>.png` + `page_map` key. Reserve `variables`/`components`/`fonts` stems. |
| 10 | **Per-stage model** | **Opus 4.6** (`claude-opus-4-6` — verify exact ID at wiring) for **all three stages**. Temps **1.0 / 0.7 / 0.6** (stage 1/2/3). |
| 11 | **Repair + batch** | Stage-3 repair **≤2 nudges/page** then drop site. Inline re-rolls **≤2/stage** then skip seed. **Generate ~48 → gate → keep 10.** Report measured gate yield (Hard-Req-#1). |
| 12 | **Curation** | Coverage auto-shortlist (distinct archetype + distinct aesthetic + complexity spread ~3/4/3) → **one human confirmation pass**. Final curation was done **by hand, audited by seed tuple**; a quantitative coverage metric is **deferred**. |

**Canonical component catalog (~20 types; stage 1's manifest draws a subset, stage 2 styles
exactly those):**
- *Chrome:* `header/nav`, `footer`
- *Atoms:* `button`, `link`, `badge/tag`, `form-field`, `icon` (inline-SVG/CSS)
- *Sections:* `hero`, `feature-grid`(+`feature-card`), `content-section`, `card`,
  `pricing-table`(+`tier-card`), `testimonial/quote`, `stat/metric`, `cta-banner`,
  `gallery/image-grid`, `logo-cloud`, `faq/disclosure` (static, open), `team/profile-card`,
  `contact-block`, `blog-post-card`, `product-card`

**Build order:** make **1 site locally and eyeball it**; once good, run the **~48-site batch
on Modal** → gate → curate 10. (No intermediate "calibration" phase.)
